# 快速掌握主题方法论

帮助用户在最短时间内建立对新主题的系统性理解，从"完全陌生"到"可以应用"。

## 核心原则

- **输出驱动**：每个阶段必须产生可见成果
- **主动建构**：学习者必须主动加工信息，而非被动接收
- **即时反馈**：快速检验理解，及时调整
- **渐进深入**：从骨架到血肉，避免一开始就陷入细节

## 学习流程

### 阶段一：目标锚定 (5-10分钟)

明确学习目标的三个维度：

1. **用途定位** - 用1句话说明学完要做什么
   - 要解决什么问题？
   - 要做出什么成果？
   - 要教会什么人？

2. **深度设定** - 明确"足够好"的标准
   - 了解层面：知道是什么、能聊起来
   - 应用层面：能动手做、能解决常见问题
   - 精通层面：能处理边界情况、能优化、能教人

3. **时间约束** - 设定硬截止日期
   - 紧急任务（< 1天）：抓核心、跳细节
   - 短期掌握（1-7天）：完整流程、适当深度
   - 深度学习（> 1周）：系统性、建立联系

**产出物**：一句话目标描述 + 深度级别 + 时间预算

### 阶段二：快速测绘 (15-30分钟)

建立主题的"地图"，不深入细节。

根据主题类型选择框架，详见 `references/learning-frameworks/`：
- 技术/工具类 → 见 `technical-topic.md`
- 概念/理论类 → 见 `conceptual-topic.md`
- 流程/技能类 → 见 `procedural-topic.md`

**核心问题**：
1. 这个主题解决什么问题？
2. 核心概念/组件有哪些？（限制在5-7个）
3. 与已知的什么知识相关？
4. 最常见的使用场景是什么？

**产出物**：主题地图（文本描述或思维导图大纲）

### 阶段三：核心突破 (时间因主题而异)

选择最关键的20%内容，掌握80%价值。

**技术类主题**：
- 搭建最小可运行环境
- 完成一个"Hello World"级别示例
- 理解核心API/命令的使用模式

**概念类主题**：
- 用自己的话定义核心概念
- 找出3个类比或例子
- 画出概念之间的关系图

**流程类主题**：
- 走完一遍完整流程
- 标记关键决策点
- 记录常见错误和解决方案

**产出物**：可运行的代码/笔记/流程记录

### 阶段四：费曼检验 (20-30分钟)

假装教给一个10岁小孩或完全的新手。

**步骤**：
1. 选择主题的一个子领域
2. 用简单语言解释，不用术语
3. 发现卡壳的地方 → 这就是知识缺口
4. 回到源头填补缺口
5. 简化类比，重新表达

**产出物**：一份教学大纲或讲解稿

使用 `references/assessment-patterns.md` 获取具体检验方法。

### 阶段五：应用固化 (持续)

通过实际应用加深理解。

**立即执行**：
- 解决一个真实小问题
- 写一个总结性的 cheatsheet
- 教给另一个人（或模拟教授）

**后续强化**：
- 24小时后回顾核心概念
- 一周后应用一次
- 一月后尝试教授他人

## 常见陷阱与对策

| 陷阱 | 表现 | 对策 |
|------|------|------|
| 完美主义 | 想从基础开始系统学习 | 用80/20法则，先抓核心 |
| 收藏癖 | 疯狂收藏资料但不看 | 限制资料源≤3个，立即开始 |
| 被动消费 | 只看视频/文章不动手 | 每看15分钟必须输出点什么 |
| 过早优化 | 纠结细节、钻牛角尖 | 先跑通主干，细节留待后续 |
| 虚假掌握 | 看懂了=会了 | 强制输出，用教来检验 |

## 与用户的交互模式

当用户请求帮助学习某个主题时：

1. **先诊断**：问清楚目标、时间、深度
2. **给框架**：根据主题类型提供学习路径
3. **陪执行**：陪伴完成5个阶段，每阶段检查产出
4. **要输出**：每个阶段必须有可见成果
5. **做调整**：根据反馈调整节奏和深度
